# Техническое задание MVP

## 1. Назначение документа

Документ определяет состав, границы и критерии приемки MVP для системы сбора, нормализации, анализа и публикации инсайтов из пользовательских и внешних источников. ТЗ предназначено для согласования с ментором, декомпозиции в задачи разработки и последующей приемки результата.

## 2. Целевой контур MVP

- Система принимает входящие данные из вебхуков, форм ввода и интеграций.
- Система выполняет нормализацию, дедупликацию, анализ текста и формирование структурированного инсайта.
- Система поддерживает ручной апрув и публикацию результата.
- Система использует FastAPI, SQLAlchemy, Pydantic, OpenAI API, Telegram Bot API и внешние API маркетплейсов.
- MVP должен быть устойчив к пустым данным, повторным запросам, rate limits и частичным отказам внешних сервисов.

# Блок А. Управление и стандарты

## A1. Паспорт инсайта и формат JSON-анализа

### 1. Технические требования

- Паспорт инсайта должен быть представлен как нормализованная сущность с обязательными полями: `insight_id`, `source`, `raw_text`, `normalized_text`, `sentiment_score`, `pain_point`, `pain_category`, `created_at`, `status`.
- `insight_id` должен быть уникальным строковым или UUID-идентификатором.
- `source` должен быть строкой ограниченного набора значений: вебхук, Telegram, маркетплейс, ручной ввод, API.
- `raw_text` и `normalized_text` должны храниться в UTF-8 и поддерживать кириллицу.
- JSON-анализ должен валидироваться через Pydantic и содержать строго определённую схему без произвольных ключей.
- Для анализа должны использоваться числовые и строковые типы: `sentiment_score` как `float` в диапазоне от `-1.0` до `1.0`, `pain_point` как краткая строка, `confidence` как `float` от `0.0` до `1.0` при наличии.
- Формат дат должен быть ISO 8601 с часовым поясом.

### 2. Логика и Edge Cases

- Если входной текст пустой, состоит только из пробелов или служебных символов, система должна отклонять запись до стадии анализа.
- Если входное сообщение содержит спам, рекламные вставки, дублирование символов или нецензурный шум, система должна помечать его как низкокачественный сигнал и понижать приоритет обработки.
- Если OpenAI возвращает невалидный JSON, пустой ответ или формат, не соответствующий схеме, система должна переходить на безопасный fallback-режим и сохранять событие ошибки.
- Если анализ частично успешен, например извлечён `sentiment_score`, но отсутствует `pain_point`, запись не должна попадать в опубликованный контур без ручной проверки.
- При повторной обработке одного и того же текста должен сохраняться детерминированный результат при одинаковых входных параметрах и версии промпта.

### 3. Инструментарий

- Pydantic для описания и валидации схемы JSON-анализа.
- FastAPI для транспортного слоя и передачи структурированных ответов.
- OpenAI API для генерации анализа.
- Стандартные средства сериализации JSON в Python.
- Логирование ошибок и трассировка ответов внешнего API через встроенный logging или структурированный логгер.

### 4. Критерии приемки (DoD)

- Ментор видит формализованную схему паспорта инсайта и JSON-анализа.
- Ментор может проверить, что невалидный JSON отклоняется на уровне Pydantic.
- Ментор видит, что пустой текст и мусорные данные не проходят в опубликованный результат.
- Ментор получает пример корректного JSON-анализа и понимает, какие поля обязательны.

## A2. Правила фильтрации дублей и таксономия болей

### 1. Технические требования

- Система должна определять дубли по комбинации `normalized_text`, `source`, `pain_point` и временной близости.
- Дедупликация должна работать на уровне входящего текста до запуска тяжелой аналитики, если это возможно.
- Таксономия болей должна быть иерархической: `категория` -> `подкатегория` -> `тег`.
- Каждая боль должна иметь человекочитаемое название, машинный идентификатор и краткое описание.
- Для каждой записи необходимо хранить ссылку на канонический инсайт и признак `duplicate_of`, если запись признана дублем.

### 2. Логика и Edge Cases

- Если несколько сообщений отличаются только пунктуацией, регистром, лишними пробелами или эмодзи, они должны рассматриваться как потенциальные дубли.
- Если смысл сообщений совпадает, но текст отличается существенно, система должна относить их к одной кластерной группе только при достаточной уверенности.
- Если текст содержит смешанные боли, система должна выбирать основную боль и дополнительно сохранять вторичные теги.
- Если дубль приходит после публикации исходного инсайта, система не должна создавать новую сущность публикации, а должна присоединить запись к уже существующему кластеру.
- Если набор классификаторов не уверен в категории, система должна использовать нейтральную группу и пометку `needs_review`.

### 3. Инструментарий

- Нормализация текста средствами Python: `re`, `unicodedata`, `str.lower()`, очистка пробелов.
- Возможные методы семантической близости: эмбеддинги, простая эвристика, либо базовые fuzzy-matching алгоритмы.
- SQLAlchemy для хранения кластеров дублей и канонических записей.
- Pydantic или Enum-классы для таксономии болей.

### 4. Критерии приемки (DoD)

- Ментор видит описанные правила, по которым одно сообщение признаётся дублем другого.
- Ментор получает список категорий болей и понимает, как они расширяются без ломки схемы.
- Ментор видит, что дубль не порождает новую публикацию и не засоряет аналитику.
- Ментор может проверить наличие `duplicate_of` и канонической записи в модели данных.

## A3. Состояния апрува и логика обратной связи

### 1. Технические требования

- Состояния апрува должны быть ограничены тремя основными статусами: `Draft`, `Pending`, `Posted`.
- Для статуса должен быть определён допустимый переход состояния и запрет на обратный переход без явной команды.
- Каждое изменение статуса должно сопровождаться timestamp, автором действия и причиной перехода.
- Обратная связь по инсайту должна храниться как отдельная сущность с полями: `insight_id`, `comment`, `author`, `created_at`, `feedback_type`.

### 2. Логика и Edge Cases

- `Draft` означает, что инсайт ещё не готов к передаче на публикацию и может быть отредактирован.
- `Pending` означает ожидание ручного подтверждения или внешнего события.
- `Posted` означает, что материал опубликован или передан в следующий интеграционный контур.
- Если ментор возвращает комментарий, статус не должен автоматически переходить в `Posted`.
- Если обратная связь противоречива или содержит несколько требований, система должна сохранять её без потери текста и маркировать как `needs_clarification`.
- Если статус изменён ошибочно, система должна иметь контролируемый путь возврата только через отдельную операцию исправления.

### 3. Инструментарий

- Enum-типизация для статусов.
- SQLAlchemy для хранения журнала переходов.
- FastAPI endpoints для смены статуса и записи feedback.
- Логирование audit trail.

### 4. Критерии приемки (DoD)

- Ментор видит явную диаграмму статусов и понимает допустимые переходы.
- Ментор может проверить, что feedback сохраняется отдельно от статуса.
- Ментор видит журнал действий по каждому инсайту.
- Ментор убеждается, что `Draft -> Pending -> Posted` реализован как контролируемый процесс, а не как свободное поле.

# Блок Б. Техническая разработка

## B1. FastAPI ядро: обработка вебхуков и асинхронность

### 1. Технические требования

- Основное приложение должно быть реализовано на FastAPI.
- Все входящие внешние события должны приниматься через webhooks или асинхронные HTTP endpoints.
- Обработчики должны поддерживать JSON body, query parameters и корректные HTTP-коды ответа.
- Асинхронная модель исполнения должна использовать AnyIO-совместимый стек.
- Для долгих операций должны применяться фоновые задачи или очереди, чтобы не блокировать HTTP-цикл.

### 2. Логика и Edge Cases

- Если вебхук приходит повторно, система должна быть идемпотентной и не создавать дубликаты действий.
- Если payload неполный, endpoint должен возвращать предсказуемую ошибку валидации, а не 500.
- Если внешний сервис присылает запрос с задержкой или повторной доставкой, обработчик должен корректно принять событие по ключу идемпотентности.
- Если обработка анализа затягивается, API не должен удерживать запрос синхронно до завершения внешней модели.
- При временной недоступности downstream-сервиса система должна сохранять событие в статусе ожидания повторной обработки.

### 3. Инструментарий

- FastAPI как HTTP-фреймворк.
- AnyIO для асинхронности и унифицированного выполнения async-задач.
- Pydantic для схем запросов и ответов.
- Uvicorn как ASGI-сервер.
- HTTP-клиент для интеграций, например `httpx`.

### 4. Критерии приемки (DoD)

- Ментор видит, что приложение поднимается как FastAPI-сервис и принимает webhooks.
- Ментор получает подтверждение, что повторный webhook не создает дубль.
- Ментор может проверить, что ошибки валидации отдаются как 4xx, а не как 500.
- Ментор видит, что долгие операции не блокируют основной HTTP-поток.

## B2. База данных: схема SQLAlchemy, индексация и миграции

### 1. Технические требования

- Модель данных должна быть описана через SQLAlchemy ORM.
- Для основных сущностей должны быть определены таблицы, первичные ключи, внешние ключи и ограничения.
- Критичные поля фильтрации и сортировки должны быть проиндексированы.
- Схема должна поддерживать хранение истории статусов, истории ответов и истории анализа.
- Миграции должны быть воспроизводимыми и не требовать ручного изменения схемы на боевом контуре.

### 2. Логика и Edge Cases

- Если запись создаётся в несколько этапов, транзакция должна сохранять консистентность или откатываться полностью.
- Если индекс отсутствует на часто используемом поле, это должно рассматриваться как дефект производительности для MVP.
- Если структура таблицы меняется, новые поля должны иметь обратимую миграцию.
- При частичной недоступности БД система должна не терять входящее событие и фиксировать его в retry-safe механизме.
- Для SQLite-режима и последующего масштабирования схема должна оставаться переносимой на PostgreSQL.

### 3. Инструментарий

- SQLAlchemy 2.x.
- Alembic для миграций.
- Session management через `sessionmaker`.
- Индексы `Index(...)` и ограничения уровня ORM/DDL.

### 4. Критерии приемки (DoD)

- Ментор видит схему таблиц и понимает назначение каждой сущности.
- Ментор может проверить наличие индексов на полях поиска и сортировки.
- Ментор получает миграции, которые можно применить на чистой базе.
- Ментор видит, что схема не завязана на ручные SQL-скрипты.

## B3. Интеграция OpenAI: gpt-4o-mini, системные промпты, RAG

### 1. Технические требования

- Для генерации анализа и текстовых ответов должна использоваться модель `gpt-4o-mini`.
- Для каждой задачи должен быть определён системный промпт с фиксированной ролью и ограничениями.
- Ответ модели должен проходить пост-валидацию на структуру и формат.
- База знаний `knowledge_base.txt` должна использоваться как источник контекстных правил в режиме RAG.
- Доступ к OpenAI API должен быть изолирован и конфигурируем через переменные окружения.

### 2. Логика и Edge Cases

- Если API недоступен, система должна использовать предсказуемый fallback, а не падать целиком.
- Если лимит запросов исчерпан, должен срабатывать режим деградации качества с сохранением работоспособности основного контура.
- Если модель возвращает невалидный JSON или лишний текст, ответ должен отклоняться и заменяться безопасной альтернативой.
- Если knowledge base пустая или отсутствует, приложение должно автоматически инициализировать её базовым набором правил.
- Если в запросе есть конфликтующие инструкции, приоритет должен быть у системного промпта и локальных бизнес-правил.

### 3. Инструментарий

- `openai` Python SDK с асинхронным клиентом.
- `gpt-4o-mini` как базовая модель MVP.
- JSON response format и локальная пост-валидация.
- `knowledge_base.txt` как простой RAG-слой начального уровня.
- Обработчики ошибок: `RateLimitError`, `OpenAIError`.

### 4. Критерии приемки (DoD)

- Ментор видит, что запросы к OpenAI выполняются через заданную модель и системный промпт.
- Ментор может проверить fallback-поведение при недоступности API.
- Ментор видит, что RAG-правила подгружаются из `knowledge_base.txt`.
- Ментор получает доказательство, что модельный ответ валидируется до записи в систему.

## B4. Коннекторы маркетплейсов: Wildberries и Ozon

### 1. Технические требования

- Интеграция должна поддерживать получение и отправку данных через API Wildberries и Ozon.
- Для каждого коннектора должны быть описаны параметры авторизации, формат запросов и формат ответов.
- Должны быть предусмотрены отдельные адаптеры под каждый маркетплейс, а не единый неразделённый клиент.
- Поля, связанные с товаром, отзывом, рейтингом и идентификатором товара, должны иметь типизацию и обязательность на уровне схемы.

### 2. Логика и Edge Cases

- При возврате 429 или аналогичного rate limit система должна применять retry/backoff и не перегружать внешний API.
- Если токен авторизации истёк, коннектор должен перевести интеграцию в состояние `reauth_required`.
- Если внешний API вернул неполный объект, система должна сохранить сырой payload и не терять диагностические данные.
- Если один маркетплейс недоступен, это не должно блокировать обработку другого источника.
- Если ответ API изменил контракт, обработчик должен детектировать несовместимость и поднять контролируемую ошибку схемы.

### 3. Инструментарий

- `httpx` или эквивалентный async HTTP-клиент.
- Retry-механизмы с backoff, например через `tenacity`.
- OAuth/API-token handling через конфигурационный слой.
- Отдельные адаптеры и DTO для Wildberries и Ozon.

### 4. Критерии приемки (DoD)

- Ментор видит отдельные коннекторы для Wildberries и Ozon.
- Ментор может проверить обработку 429 и повторных попыток.
- Ментор видит, что ошибка одного коннектора не валит весь сервис.
- Ментор получает пример сохранённого сырого ответа для диагностики.

## B5. Telegram Bridge: уведомления в конкретный thread_id группы

### 1. Технические требования

- Telegram Bridge должен отправлять уведомления в заданную группу и конкретный `thread_id`.
- Сообщение должно содержать минимально достаточный набор данных: заголовок, краткое содержание инсайта, статус и ссылку на объект.
- Формат сообщения должен быть предсказуемым и пригодным для ручной модерации.
- Необходимо поддерживать конфигурацию chat_id, thread_id и токена бота через env-переменные.

### 2. Логика и Edge Cases

- Если thread_id недоступен или некорректен, система должна зафиксировать ошибку маршрутизации и не терять сообщение.
- Если Telegram API возвращает rate limit, должна быть предусмотрена повторная отправка по безопасной схеме.
- Если группа недоступна, сообщение должно уходить в очередь повторной доставки или в журнал неотправленных уведомлений.
- Если текст уведомления превышает ограничения Telegram, он должен быть сокращён или разбит на логические части.
- Если бот лишён прав на отправку в thread, событие должно маркироваться как `delivery_failed`.

### 3. Инструментарий

- Telegram Bot API.
- `python-telegram-bot` или прямые HTTP-запросы через `httpx`.
- Очередь повторной отправки и журнал доставки.
- Шаблонизация текста уведомления.

### 4. Критерии приемки (DoD)

- Ментор видит, что уведомления отправляются именно в нужный thread_id.
- Ментор может проверить поведение при rate limit и ошибке доступа.
- Ментор получает подтверждение, что неудачная доставка не теряет событие.
- Ментор видит, что формат сообщения пригоден для ручного чтения в группе.

# 3. Общие нефункциональные требования

- Архитектура должна быть расширяемой по источникам данных и каналам доставки.
- Все критичные операции должны логироваться.
- Все внешние интеграции должны иметь безопасный fallback.
- Схемы данных должны быть типизированы и валидируемы.
- Система должна сохранять поведение при частичной деградации внешних сервисов.
- Для MVP допускается упрощённая реализация, но без потери контрактов данных и управляемости статусов.

# 4. Общий критерий готовности MVP

MVP считается готовым, если ментор может:

1. Принять входной инсайт и увидеть его паспорт в структурированном виде.
2. Проверить дедупликацию и таксономию болей.
3. Пройти путь статусов `Draft -> Pending -> Posted`.
4. Увидеть стабильную обработку вебхуков и асинхронных задач.
5. Проверить работу БД, миграций, OpenAI-интеграции, маркетплейс-коннекторов и Telegram Bridge.
6. Убедиться, что система не падает на пустых данных, дублях, rate limits и временных сбоях внешних API.
